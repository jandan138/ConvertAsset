vMaterials 与 MDL SDK 生态

MDL 并非孤立存在。NVIDIA 围绕它建立了一个完整的生态：用于运行时集成的 MDL SDK、提供数千种物理材质的 vMaterials 库、以及 Omniverse / Isaac Sim 平台内的一系列配套工具。了解这个生态，能帮助你理解为什么现实中的 Isaac Sim 资产充满了 MDL 材质，以及转换工作面对的实际复杂度。

vMaterials：MDL 材质库

vMaterials 是 NVIDIA 官方发布的物理材质库，包含金属、塑料、木材、皮革、织物、玻璃等数百种（累计超过 1000+）真实感材质，每一种都以 .mdl 文件的形式提供。这些材质的特点：

  • 基于真实测量数据（反射曲线、SVBRDF）构建，物理精度高。
  • 大量使用 MDL 的 BSDF 组合算子——例如车漆会用多层 fresnel_layer + backscattering_glossy_reflection_bsdf，普通的 Cook-Torrance 参数根本无法精确表达。
  • 配套高分辨率 PBR 纹理包（diffuse / normal / roughness / metallic / AO 等）。
对 ConvertAsset 的影响 Isaac Sim 内置的 SimReady 资产普遍使用 vMaterials 系材质——即 OmniPBR.mdl、OmniGlass.mdl、OmniSurface.mdl 等 Omniverse MDL 扩展。当你用 ./scripts/isaac_python.sh ./main.py no-mdl 处理这些资产时，no_mdl/materials.py 需要从这些 MDL 的 inputs 里提取 BaseColor / Roughness 等参数，近似映射到 UsdPreviewSurface。

MDL SDK：运行时集成

MDL SDK 是 NVIDIA 提供的 C++ 库，提供：

  • 编译与 JIT：把 MDL 材质编译为 LLVM IR / CUDA PTX，可以在 GPU 上求值 BSDF。
  • 参数访问：通过反射机制列举材质的 inputs（类型、名称、默认值、注解），这是工具链提取参数的基础。
  • Distilling API：将任意 MDL 材质"蒸馏"（distill）为指定的目标模型（如 ue4 或 transmissive_pbr），自动折叠 BSDF 树，得到近似的高层参数——这正是 no-MDL 转换的理论基础之一。
ConvertAsset 的实际策略 ConvertAsset 当前不依赖 MDL SDK。它通过直接解析 USD Shader prim 的 inputs: 字段（以及可选的 .mdl 文件文本），静态提取 BaseColor / Roughness / Metallic / Normal 等参数，然后手工构建 UsdPreviewSurface 网络。这个策略的优点是无需集成 MDL SDK（降低依赖复杂度）；代价是对复杂 BSDF 组合材质的近似精度有限，mdl_parse.py 通过正则表达式提取参数。

MDL 在 Omniverse 生态中的地位

在 Omniverse 架构中，MDL 扮演了材质的"黄金标准"：

  • Isaac Sim：默认使用 OmniPBR / OmniSurface MDL，配合 RTX 渲染器实现高保真仿真渲染。
  • Omniverse Create / Composer：材质编辑器的底层就是 MDL 节点图，导出时自动生成 .mdl 文件。
  • USD 材质绑定：每个 Mesh Prim 通过 material:binding 关系绑定到一个 Material Prim，Material Prim 同时持有 outputs:mdl:surface（MDL 路径）和 outputs:surface（UsdPreviewSurface 路径）。理想情况下两者并存，提供 portability；但很多资产只有 MDL 路径。
第 5 章小结 MDL 是一个设计精良的材质语言：声明式物理接口 + 过程式函数 + 完备的 BSDF/EDF/VDF 体系 + 模块化库（vMaterials）。但它的执行依赖 MDL SDK，形成事实上的 renderer-bound 约束。这个约束催生了第 6 章的 no-MDL 转换——在保留 USD 组合结构的前提下，把 MDL 材质网络近似映射到 UsdPreviewSurface，换取 portability。
