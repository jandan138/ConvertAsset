为什么 renderer-bound

MDL 的设计理念是"渲染器无关"（renderer-agnostic）——声明式材质结构只描述物理属性，不依赖任何具体的渲染算法。然而在实践中，MDL 是显著的 renderer-bound（渲染器绑定）：没有集成 MDL SDK 的渲染器根本无法消费它。理解这个矛盾，是理解 ConvertAsset 存在理由的前提。

MDL SDK 依赖链

MDL 文件不是自解释的着色代码。要让渲染器"理解"一个 .mdl，需要完整的处理链：

  1. 解析（parsing）——MDL SDK 的词法器/语法器把 MDL 文本解析为内部 AST。
  2. 链接（linking）——解析 import 语句，把所有依赖的模块（包括 NVIDIA 标准库 ::df、::tex 等）链接进来。
  3. 编译（compilation）——MDL SDK 通过 LLVM / PTX 后端进行 JIT 编译，把材质函数编译成能在目标硬件（CUDA / CPU）上运行的代码。
  4. 运行时求值——在光线追踪时，每个 shading point 都调用编译好的 BSDF evaluate / sample 函数。

这条链路完全依赖 NVIDIA 的 MDL SDK（一个闭源的 C++ 库）。任何一个环节缺失，材质就无法渲染。这就是 renderer-bound 的根本原因：不是 MDL 语言设计的问题，而是执行基础设施的问题。

与 GLSL / OSL 的对比 GLSL 和 Metal Shading Language 是 GPU 驱动直接支持的着色语言；OSL 虽然也需要编译，但有 Intel / Blender / Arnold 等多方开源实现。MDL 的主要实现仅 NVIDIA 官方的 MDL SDK，事实上是供应商锁定（vendor lock-in）。

MDL 在 USD 里的表达方式

在 NVIDIA Omniverse / Isaac Sim 中，USD 材质通过 UsdShade 网络表达。一个使用 MDL 的材质 Prim 看起来像这样：

mdl_in_usd.usda
def Material "M_Wood" {
    # MDL shader prim
    def Shader "MDLShader" {
        uniform token info:implementationSource = "sourceAsset"
        uniform asset info:mdl:sourceAsset = @OmniPBR.mdl@
        uniform token info:mdl:sourceAsset:subIdentifier = "OmniPBR"

        # MDL 参数（通过 inputs: 前缀传递）
        float inputs:reflection_roughness_constant = 0.4
        color inputs:diffuse_color_constant = (0.6, 0.45, 0.3)
        asset inputs:diffuse_texture = @textures/wood_basecolor.png@
    }

    # MDL 的 outputs 连接
    token outputs:mdl:surface.connect = 
    token outputs:mdl:volume.connect  = 
}

注意 outputs:mdl:surface 这个端口——它是 MDL 专属的，任何不支持 MDL 的工具（如标准 usdview、Blender USD importer、Web viewer）都会忽略它，导致材质完全失效或显示为默认灰色。

为什么这很重要 在机器人仿真场景中，资产既要在 Isaac Sim（支持 MDL / RTX）里高保真渲染，也要能被下游的 Web viewer、VLM 标注工具、GLB 出口管线等非 NVIDIA 工具消费。portability 是刚需，而非锦上添花——这正是第 6 章 no-MDL 转换的出发点。
