转换动机：portability vs fidelity

一个 Isaac Sim 仿真场景里的机器人和家具，往往拥有精心制作的 MDL 材质——它们在 RTX 渲染器下看起来逼真，物理上忠实。然而一旦需要把这个场景导出成 GLB 给 Web viewer 看，或者让 VLM（视觉语言模型）处理渲染图，MDL 材质就成了障碍：工具看不懂它，资产变成一片灰。portability 与 fidelity 的张力，是 ConvertAsset 整个项目的出发点。

portability 与 fidelity 的根本张力

这个张力可以用一句话概括：越忠实物理现实的材质表示，越难做到跨工具可用。

  • 高 fidelity 路径：MDL + RTX 渲染器。BSDF 组合层次完整，次表面散射、各向异性光泽、精确 Fresnel 全部保留。代价：只能在 NVIDIA 生态里渲染，其他工具完全看不懂。
  • 高 portability 路径：UsdPreviewSurface / glTF metallic-roughness。几乎所有 USD-aware 工具、Web 引擎、VLM 输入管线都支持。代价：被降维到 Cook-Torrance metallic-roughness 参数化，复杂 MDL 效果无法精确表达。
Isaac Sim 场景的典型材质结构 Isaac Sim 内置的 SimReady 资产一般同时包含两个材质端口：outputs:mdl:surface（MDL 路径）和 outputs:surface（UsdPreviewSurface 路径）。理想情况下两者并存，工具按能力选择。但现实中大量资产只有 MDL 路径，导致非 NVIDIA 工具看到的是空白材质。no-MDL 转换就是补上缺失的那半边。

MDL 的 renderer-bound 问题回顾

MDL 的执行完全依赖 NVIDIA MDL SDK（一个 C++ 闭源库）。没有集成这个 SDK 的渲染器或工具无法消费 MDL 材质。具体影响：

  • 标准 usdview（OpenGL/Vulkan 模式）——显示为默认灰色。
  • Blender USD importer——忽略 outputs:mdl:surface，材质失效。
  • Web-based USD viewer / Three.js——不支持 MDL，材质缺失。
  • GLB 导出（export-glb 子命令）——只能读 UsdPreviewSurface，没有它就导出全灰 GLB。
  • VLM 渲染图采集（用于 grounding 评测）——如果用不支持 MDL 的渲染器渲染，材质全灰，与真实场景的视觉分布差异很大。

no-MDL 转换做什么

ConvertAsset 的 no-mdl 子命令（入口：convert_asset/no_mdl/processor.py）做的事情是：

  1. 对每一个 USD 文件，在同目录生成一个 *_noMDL.usd 兄弟文件（SUFFIX = "_noMDL"）。
  2. 在新文件里，把每个 Material 的 MDL Shader 网络替换为 UsdPreviewSurface 网络（尽量复用贴图和参数）。
  3. 重写所有 references / payloads / sublayers 的路径，让它们指向对应的 *_noMDL.usd。
  4. 不 flatten——USD 的组合结构（layers / variants / references）完整保留，只改路径和材质网络。
什么是"合理近似" no-MDL 转换的输出并不等于 MDL 的精确复现。它是一个有损近似（lossy approximation），目标是"在 portability 工具链下视觉上合理，下游任务可用"——而不是"数值上等价"。后续 6-4 节会从论文视角讨论这个近似的量化边界。
