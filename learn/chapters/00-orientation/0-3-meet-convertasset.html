认识 case study：ConvertAsset 与它的论文问题

ConvertAsset 是一个运行在 Isaac Sim Python 环境里的命令行工具，做三件事：把 MDL 材质转换成 UsdPreviewSurface（no-mdl）、用 QEM 简化 mesh（mesh-simplify）、导出 GLB（export-glb）。它的研究问题是：这些「必要的格式化简」是否会改变下游 AI 模型对资产的理解？这节把工具本身走一遍，然后解释为什么这个问题值得研究。

CLI 快速浏览

所有命令都通过 Isaac Sim 的 Python wrapper 执行。ConvertAsset 的核心 subcommand 如下：

terminal
# wrapper 自动探测 Isaac Sim 安装位置
./scripts/isaac_python.sh ./main.py <subcommand> [args]

# 1) MDL → UsdPreviewSurface（最常用，也是默认子命令）
./scripts/isaac_python.sh ./main.py no-mdl /path/to/scene.usd

# 2) mesh 简化（--ratio 控制保留比例）
./scripts/isaac_python.sh ./main.py mesh-simplify /path/to/scene.usd \
    --backend py --ratio 0.5 --apply --out /output.usd

# 3) 导出 GLB
./scripts/isaac_python.sh ./main.py export-glb /path/to/scene_noMDL.usd \
    --out /output.glb

# 4) 一键完整 pipeline（no-mdl + mesh-simplify + export-glb）
./scripts/isaac_python.sh ./main.py usd-to-glb /path/to/scene.usd \
    --out /output.glb

架构：三个关键设计决策

ConvertAsset 的代码组织反映了三个有意识的工程决策，本书后续章节会逐一深入：

决策 1：composition-preserving（不 flatten）

no-mdl 转换采用非扁平化策略：对每个 USD 文件独立处理，生成一个同目录的 *_noMDL.usd 兄弟文件，保留全部 composition arcs（references / payloads / variants / sublayers），只改写 assetPath 指向新文件、转换材质节点。

这与「先 flatten 再处理」的方案相比，保留了资产结构的可审计性：你仍然可以在 usdview 里展开层叠结构，逐级查看哪一层的意见最终生效。实现这一策略的核心是 Processor 类：

convert_asset/no_mdl/processor.py（节选）
class Processor:
    """递归处理 USD，生成对应的 *_noMDL.usd。

    成员：
    - done: dict[str, str]
        已处理过的"源绝对路径 -> 目标 *_noMDL 绝对路径"映射，用于去重和最终汇总。
    - in_stack: set[str]
        当前递归栈中的源路径集合，用于检测环状引用，避免无限递归。
    """
    def __init__(self):
        self.done = {}       # src_abs -> dst_abs（去重）
        self.in_stack = set()  # 环检测

    def process(self, src_usd_abs: str, _is_root_call: bool = True) -> str:
        # 快速返回：已处理过直接复用结果
        if src_usd_abs in self.done:
            return self.done[src_usd_abs]
        # 环检测：若在递归栈中，提示并返回已有结果
        if src_usd_abs in self.in_stack:
            print("[WARN] cyclic reference? already in stack:", src_usd_abs)
            return self.done.get(src_usd_abs, src_usd_abs)
        self.in_stack.add(src_usd_abs)
        # ... 递归处理所有子 USD，然后处理当前文件 ...
两个集合各司其职 done（dict）防止重复处理同一文件——一个 asset 被多个场景引用时，只处理一次。in_stack（set）检测循环引用（A 引用 B，B 又引用 A 的情况），防止无限递归。两者协作构成了整个工具的安全保障。

决策 2：lazy imports（懒加载 pxr）

pxr（USD Python 绑定）是 Isaac Sim 提供的重量级依赖，在普通 Python 环境里无法 pip install。ConvertAsset 的所有模块把 from pxr import ... 放在函数体内部，不放在文件顶层。这样：①测试用的 mock 可以在 import 时注入；②工具能在 pxr 不可用的环境下加载 CLI 帮助，而不立即崩溃。

决策 3：多 backend mesh simplification

mesh simplification 支持三种 backend：

  • py：纯 Python QEM 实现，无需编译，速度较慢。
  • cpp：调用编译后的 C++ 可执行文件（subprocess），速度快。
  • cpp-uv：pybind11 原地调用，速度最快，且保留 UV。

C++ backend 是可选的；工具在没有 C++ 编译环境时用纯 Python 完整运行。

论文问题：conversion as controlled perturbation

整个工具背后有一个严肃的研究问题：

核心研究问题 MDL → UsdPreviewSurface 的材质转换，以及 high-poly → low-poly 的 mesh 简化，必然引入外观差异（controlled perturbation）。这些差异是否会影响下游 VLM 模型的 visual grounding 能力？换句话说：资产转换对 AI「看」这个资产的方式，影响有多大？

这个问题的研究框架是：

  1. 用同一个资产，分别渲染「原始 MDL」版本和「转换后 UsdPreviewSurface」版本。
  2. 用 PSNR / SSIM / LPIPS 等感知度量量化外观差异。
  3. 用 CLIP / DINOv2 等语义特征量化 AI 视觉表示的差异。
  4. 在 VLN（Vision-Language Navigation）任务上测试：不同版本渲染出的场景，AI 完成导航任务的成功率差多少？

这种「把工程必要性与研究可量化性对齐」的设计，正是本书用 ConvertAsset 作为 case study 的原因——它不只是一个好用的工具，它还是一个有清晰 evidence gate 的科学实验框架。

From GAMES101 你在 GAMES101 里用 PSNR 来衡量图像压缩质量——ConvertAsset 用完全相同的度量来衡量材质转换引入的外观偏差。唯一的区别是：这里的「参考图」是 MDL 渲染结果，「测试图」是 UsdPreviewSurface 渲染结果，而不是原图与压缩图。

本书如何使用 ConvertAsset

本书把 ConvertAsset 作为贯穿全书的 case study，在每个重要知识点上通过它的真实代码来验证理论：

  • 第 2 章（Composition）：为什么 Processor 不 flatten，而是保留 composition arcs？
  • 第 5 章（MDL）：MDL 是什么，为什么它「renderer-bound」，为什么需要转换？
  • 第 6 章（no-MDL 算法）：材质节点网络如何逐属性翻译为 UsdPreviewSurface？
  • 第 8 章（QEM）：Quadric Error Metric 推导，以及 UV 保持的工程细节。
  • 第 10 章（USD→GLB pipeline）：face-varying UV 展平，glTF binary 写出。

你不需要现在就跑通代码。只需要建立这样一个 mental model：ConvertAsset 是一棵真实的树，本书每一章都在它的某个分支上停下来，用理论解释那里发生了什么。
