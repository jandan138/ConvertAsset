你的 GAMES101 渲染器到哪为止

GAMES101 教会你渲染的底层逻辑：从光栅化到路径追踪，从 Blinn-Phong 到 Cook-Torrance。这些知识是真实的，扎实的。但课程结束时，你手里有一个「单文件软件渲染器」——而工业界每天处理的是成千上万个互相引用的资产文件、材质库与动画层。这一节帮你画出课堂渲染器的边界，再看清边界外面是什么。

GAMES101 覆盖了什么

如果你认真完成了 GAMES101，你的代码里已经有这些能力：

  • 几何处理：从 OBJ 文件读顶点与三角形，应用 model / view / projection 变换，得到屏幕坐标。
  • 光栅化：rasterization 扫描线填充，z-buffer 深度测试，bilinear 纹理过滤，mipmap。
  • 着色模型：Blinn-Phong local illumination；BRDF 概念；基于 microfacet 理论的 Cook-Torrance specular；粗糙度与金属度参数。
  • 全局光照：path tracing 基础，Monte Carlo 积分，Russian Roulette 终止。
  • 场景表示：一个 scene graph 节点树（通常是手写的、固定的），每个节点含 transform 和 mesh。
你已经有的最重要的东西 对渲染方程（rendering equation）的直觉：光如何在物体表面弹射、能量如何守恒。这是理解 production material 系统（PBR、MDL、UsdPreviewSurface）的底层基础，本书会反复回来用它。

边界在哪里

GAMES101 渲染器有意简化了若干工程复杂度，在真实 production 中这些恰恰是最难的部分：

场景表示：一个文件 vs 数千个互引文件

GAMES101 通常从单个 OBJ/FBX 加载整个场景。而 production 的场景是一张有向图：一个「主场景文件」引用「环境文件」，环境文件引用「家具文件」，家具文件引用「材质库文件」……每一层都可能有 variant（金色版 / 木纹版），每一层都可能被多个项目复用。你的 OBJ loader 对此一无所知。

材质系统：硬编码 vs 声明式节点网络

GAMES101 里你用 C++ 或 Python 把着色公式硬编码在 shader 里。Production 资产的材质是数据，以节点网络的形式写在文件里（MDL、UsdPreviewSurface、glTF metallic-roughness）。渲染引擎读取这些数据，动态编译或解释。这带来了可移植性——但也带来了「不同渲染器用不同材质格式」的兼容性地狱。

规模与协作：单机 vs 工作室

你的渲染器是一个人写的单体程序。Production pipeline 里，建模师、材质师、动画师各自工作在同一资产的不同文件层上，互不覆盖，最后由合成系统合并。这需要一套精确的「谁的意见更强」规则。

交付格式：内部格式 vs 通用 interchange

你的渲染器只需要读自己写的格式。但 production pipeline 末端要把资产交给游戏引擎（Unity / Unreal）、仿真平台（Isaac Sim）、AI 训练框架——每一个都有自己偏好的格式。USD 和 glTF 是两个最重要的 interchange 标准。

From GAMES101 你在 GAMES101 里写过 scene graph（节点树 + transform）。USD 的 Stage 本质上也是一个 scene graph，只是它活在多个文件里，支持非破坏性覆盖与层叠。从你熟悉的「节点树」出发，沿着「如果这棵树可以跨文件、跨版本非破坏性编辑，会怎么设计」这条线思考，你就在向 USD 靠近。

建立新的 mental model

本书不打算推翻 GAMES101 教给你的东西——光栅化、BRDF、路径追踪都是真实且必要的知识。本书做的事情是在你已有的知识体系上，搭一层新的脚手架：

  • 用 USD 替换「单文件场景」——学会读懂 Stage / Prim / Layer / Composition。
  • 用 UsdPreviewSurface 和 glTF metallic-roughness 替换「硬编码 BRDF」——理解声明式材质节点网络。
  • 用 asset pipeline 替换「单体渲染器」——理解 MDL → UsdPreviewSurface 转换、mesh simplification、GLB 导出的全链路。
