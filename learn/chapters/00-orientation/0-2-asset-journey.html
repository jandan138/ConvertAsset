production asset pipeline 全景：the asset journey

一个 3D 资产从建模师的工作站出发，到最终被 AI agent 在仿真环境里「看见」，要经历六个阶段。每个阶段都有格式转换、有信息损失、有工程取舍。ConvertAsset 聚焦在中间三个阶段：no-MDL conversion、mesh simplification、GLB export。这一节把全链路走一遍，让你看清每个环节在做什么，以及为什么需要做。

From GAMES101 你在 GAMES101 里看到的场景通常是「一个 OBJ + 一张纹理贴图」。Production 资产的起点是一个复杂的 USD Stage，里面有数百个互相引用的文件、多套材质库、数个 variant set。但最终它也要变成「顶点 + 三角形 + 材质参数」——你在 GAMES101 里处理的那种格式。整个 journey 就是在做这个「从复杂组合到运行时格式」的化简。

六个阶段一览

深色框是 ConvertAsset 直接处理的三个阶段；白框是上游输入和下游消费者。

每个阶段在做什么

① Authored USD（上游输入）

资产作者（可能是 NVIDIA 的 SimReady 资产库，也可能是工作室自制）在 Omniverse Composer 或 Maya/Houdini 的 USD 插件里创作资产。输出是一组互相引用的 USD 文件（.usd/.usda/.usdc）。材质描述用 MDL（NVIDIA Material Definition Language）——这是 Omniverse / Isaac Sim 的 native 材质格式，渲染效果出色，但只有 NVIDIA 渲染器理解它。

② no-MDL conversion（ConvertAsset 第一步）

MDL 材质无法直接在非 NVIDIA 渲染器里使用，也无法导出为 glTF。no-MDL conversion 把 MDL 节点网络翻译成 UsdPreviewSurface——USD 标准的、跨渲染器可移植的 PBR 材质描述。整个过程保持 USD composition 结构不变，只修改材质节点，并把所有文件路径重写为新的 *_noMDL.usd 兄弟文件。

③ mesh simplification（ConvertAsset 第二步）

高细节（high-poly）mesh 用于渲染出图是合理的，但送进仿真引擎或 AI 模型时往往过重。mesh simplification 用 QEM（Quadric Error Metric）算法在给定错误容限内缩减三角形数量，同时尽量保持 UV 坐标和外观。ConvertAsset 提供三种 backend：纯 Python（py）、C++ 子进程（cpp）、pybind11 原地调用（cpp-uv）。

④ export GLB（ConvertAsset 第三步）

把经过材质转换与简化的 USD 导出为 GLB（binary glTF 2.0）。GLB 是游戏引擎与 web viewer 的通用格式：单文件、包含 mesh 二进制数据与 PBR 材质参数、体积小、加载快。这一步需要把 USD face-varying UV 展平成 glTF 要求的 per-vertex UV。

⑤ render / Isaac Sim

GLB（或直接用 noMDL USD）被送进 NVIDIA Isaac Sim，用 RTX 渲染器出图。Isaac Sim 同时提供物理仿真（PhysX）和 RTX 光追渲染，是机器人学与具身 AI 的主流仿真平台。

⑥ downstream AI（下游消费者）

渲染图像作为训练数据送进 VLM（Vision-Language Model）/ VLN（Vision-Language Navigation）任务。此时材质转换引入的外观差异（controlled perturbation）是否影响 AI grounding 能力，正是本书 case study 的核心研究问题。

数据格式的演变

理解整条 pipeline 的关键是跟踪「数据在每个阶段是什么格式、损失了什么」：

asset_journey_formats.txt
# 各阶段的数据格式与关键特性
阶段 1: Authored USD
  格式: .usd / .usda / .usdc (+ 嵌套引用的子 USD)
  材质: MDL（NVIDIA 私有，高保真度）
  mesh: 高多边形，face-varying UV，subdivision surface
  组合: 完整 composition arcs（sublayers / references / payloads / variants）

阶段 2: noMDL USD (*_noMDL.usd)
  格式: .usd（与原始文件同目录，sibling）
  材质: UsdPreviewSurface（跨渲染器标准 PBR）
  mesh: 不变（仅路径改写）
  组合: 保持原始 composition 结构

阶段 3: simplified USD
  格式: .usd（--out 指定路径）
  材质: UsdPreviewSurface（不变）
  mesh: 简化后三角形数减少（ratio 可配置）

阶段 4: GLB (*_noMDL.glb)
  格式: .glb (binary glTF 2.0，单文件)
  材质: glTF metallic-roughness（PBR 参数内嵌）
  mesh: 三角形 + per-vertex UV（face-varying 已展平）
  组合: 扁平化（无 USD composition，所有数据内联）
glTF 要求扁平化 glTF 不支持 USD 的 composition arcs。export-glb 步骤会把 USD 的层叠结构完全展平（flatten）——这是一次有损的、不可逆的操作。因此本书的 case study 特别强调：在 USD 阶段（no-MDL、mesh-simplify）都应保持 composition 结构，只有最终交付给不理解 USD 的消费者时才做 flatten。
