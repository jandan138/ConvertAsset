case：为什么 ConvertAsset 选择 composition-preserving

把一个有 MDL 材质的 USD 场景转换成 UsdPreviewSurface，最简单的实现是「flatten」：把整个 composition 解析后一次性写成一个单层 USD 文件，然后替换材质。但 ConvertAsset 没有这样做——它选择了更复杂的 composition-preserving 路径，逐文件递归处理，保留所有 composition arc。这个设计决策背后有什么考量？代码里是怎么实现的？

Flatten 的代价

USD 提供 stage.Flatten() API：把所有 Layer 的 opinion 合并计算，输出成一个单文件「快照」。这样做有几个代价：

  • 丢失 composition 结构：references / payloads / variant sets 全部消失，变成一坨平铺的 Prim 数据。flatten 后无法再切换 variant，无法再 override 单个 sublayer。
  • 文件膨胀：原本「chair.usd 被 10 个场景 reference」只需存一份 chair 的数据；flatten 后每个场景都有一份 chair 的完整数据。
  • 可调试性下降：flatten 后你看不到「这个属性的值来自哪个 Layer，是哪个 variant 决定的」；调试材质问题失去了上下文。
  • 不可逆：flatten 是单向操作，无法从 flatten 结果反推原始 composition 结构。
对 Isaac Sim / Omniverse pipeline 尤为关键 Isaac Sim 场景通常有多层引用：顶层 scene.usd → room.usd → furniture.usd → material_library.usd。如果 flatten 处理，场景里每一件家具的材质数据都会被内联一份。一个包含 100 件家具的房间，flatten 后文件大小可能从几 MB 暴增到几百 MB，且无法再单独更新 material_library。

Processor 的设计

ConvertAsset 用 Processor 类实现 composition-preserving 转换。它的核心思路是：

  1. 对当前 USD 文件，扫描所有 composition arc 里引用的子 USD 路径。
  2. 先递归处理所有子文件，得到「子文件原路径 → 子文件 noMDL 路径」的映射 mapping。
  3. 导出当前文件的 root layer（不含子文件内容）为 *_noMDL.usd。
  4. 在新文件上把所有 assetPath 按 mapping 改写（chair.usd → chair_noMDL.usd）。
  5. 仅在当前文件上做材质转换（不涉及子文件）。

关键点：步骤 3 导出的是 root layer 本身（通过 GetRootLayer().Export()），不是 flatten 后的内容。所有子文件的内容仍然通过 composition arc 间接引入，只是路径改写了。

processor.py（核心流程）
def process(self, src_usd_abs: str, _is_root_call: bool = True) -> str:
    # 1. 规范化路径
    src_usd_abs = _to_posix(os.path.abspath(src_usd_abs))

    # 2. 去重：已处理过直接返回
    if src_usd_abs in self.done:
        return self.done[src_usd_abs]

    # 3. 环检测：防止 A→B→A 这样的循环引用导致无限递归
    if src_usd_abs in self.in_stack:
        print("[WARN] cyclic reference? already in stack:", src_usd_abs)
        return self.done.get(src_usd_abs, src_usd_abs)
    self.in_stack.add(src_usd_abs)

    # 4. 打开源 Stage
    stage = Usd.Stage.Open(src_usd_abs)

    # 5. 收集所有子 USD 路径（references / payloads / sublayers / variants / clips）
    deps = _collect_asset_paths(stage)
    child_abs_paths = set()  # 用 set 去重，同一子文件可被多处引用
    for kind, holder, layer_dir, assetPath, prim_path, extra in deps:
        abs_child = _resolve(layer_dir, assetPath)
        if ext in (".usd", ".usda", ".usdc", ".usdz"):
            child_abs_paths.add(abs_child)

    # 6. 先递归处理所有子文件，建立 mapping
    mapping = {}
    for child in sorted(child_abs_paths):
        if os.path.exists(child):
            dst_child = self.process(child, _is_root_call=False)
            mapping[child] = dst_child

    # 7. 导出当前 root layer（不 flatten！）为 *_noMDL.usd
    dst_usd_abs = _sibling_noMDL_path(src_usd_abs)
    stage.GetRootLayer().Export(dst_usd_abs)  # 只导出当前 layer 自己的数据

    # 8. 打开新文件，改写 assetPath + 转换材质
    dst_stage = Usd.Stage.Open(dst_usd_abs)
    _rewrite_assets_in_stage(dst_stage, mapping)           # 路径改写
    convert_and_strip_mdl_in_this_file_only(dst_stage)     # 材质转换

    # 9. 保存
    dst_stage.GetRootLayer().Export(dst_usd_abs)

    # 10. 记录结果、出栈
    self.done[src_usd_abs] = dst_usd_abs
    self.in_stack.remove(src_usd_abs)
    return dst_usd_abs

两个安全机制：done 与 in_stack

Processor 有两个成员变量，各自解决一个问题：

done: dict[str, str] — 去重

production USD 场景里，同一个 asset 文件往往被多处引用：同一张椅子放在同一个房间的 10 个位置，或者同一份材质库被 50 个场景引用。

如果没有 done，Processor 会对 chair.usd 处理 10 次，生成 10 次（幂等的，但完全浪费时间）。有了 done，第一次处理完后把 chair.usd → chair_noMDL.usd 记录在 done 里，后续 9 次都直接返回缓存结果。

in_stack: set[str] — 环检测

USD 技术上允许循环引用（A 引用 B，B 又引用 A），虽然这通常是错误，但如果 Processor 不处理，就会无限递归直到栈溢出。

in_stack 在进入每个文件时入栈（add），处理完成后出栈（remove）。在处理某个文件的过程中，如果递归路径再次遇到同一个文件，说明检测到了循环——发出 [WARN] 并返回已有结果（或原路径），打断循环。

cycle_detection.py（演示）
# 假设 scene.usd 引用 room.usd，room.usd 又（错误地）引用 scene.usd
# 调用栈：
#   process("scene.usd")
#     in_stack = {"scene.usd"}
#     → process("room.usd")
#         in_stack = {"scene.usd", "room.usd"}
#         → process("scene.usd")  ← 在 in_stack 里！
#             [WARN] cyclic reference? already in stack: scene.usd
#             return done.get("scene.usd", "scene.usd")  ← 打断循环
#         in_stack = {"scene.usd"}
#     in_stack = {}
# 处理完成，没有无限递归

为什么这对 production pipeline 重要

Composition-preserving 策略在 ConvertAsset 的使用场景里有三个直接价值：

  1. 可审计性：转换后的 *_noMDL.usd 仍然可以在 usdview 里展开，逐层查看 composition 结构，验证每一个材质转换是否正确。如果 flatten，整棵层叠树就变成不透明的扁平数据。
  2. 增量更新：如果只有某一个子 USD 文件的材质需要重新转换，只需要重新处理那一个文件，其他文件的 *_noMDL 版本仍然有效。flatten 策略下整个场景需要重新处理。
  3. 保持 variant 功能：noMDL 版本的场景仍然支持 variant 切换，下游用户可以在更强的 Layer 里切换椅子颜色、桌子材质——不会因为 flatten 而丢失这个能力。
本章总结 你现在理解了为什么 ConvertAsset 需要递归处理整棵 USD 引用树（composition arc 把场景分布在多个文件里），为什么 done 和 in_stack 是必要的（去重 + 环检测），以及为什么用 GetRootLayer().Export() 而不是 stage.Flatten()（保持 composition 结构的可审计性与可维护性）。这些设计决策直接来自本章讲解的 USD composition 机制。
