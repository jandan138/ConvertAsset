references vs payloads（延迟加载）

reference 和 payload 都是「把外部 USD 内容引入当前 Prim」的 arc，在 composition 语义上几乎相同。关键区别只有一个：payload 的内容可以延迟加载（defer loading）——Stage 打开时先跳过，等你真正需要时再 load。这个小小的区别，是处理 GB 级场景的关键工程决策。

From GAMES101 在 GAMES101 里你一次性把所有 mesh 读进内存，然后渲染。这在小场景可行。但一个包含 10,000 件道具的城市场景，如果每件道具都是一个独立的高多边形 USD 文件，全部加载可能需要几十 GB 内存。payload 的「延迟加载」机制让你先看到场景骨架，再按需拉取数据——就像网页的懒加载图片。

reference：立即加载

使用 references arc 引入的内容，在 Stage 打开时立即加载。这意味着：

  • Stage 一旦打开，所有 referenced 文件的内容就进入内存。
  • 遍历时能立即拿到 referenced Prim 的所有属性值。
  • 适合轻量资产或核心资产——打开场景必须马上用到的东西。
reference_example.usda
#usda 1.0
def Xform "Scene" {
    # reference：打开 Stage 时立即加载 table.usd 的全部内容
    def Xform "Table" (
        references = @./assets/table.usd@
    ) { }

    # 也可以引用某个 Prim 的路径（不用 defaultPrim）
    def Xform "Chair" (
        references = @./assets/furniture.usd@
    ) { }
}

payload：延迟加载

使用 prepend payload arc 引入的内容，默认在 Stage 打开时不加载——Stage 知道这个 Prim 存在（能看到它的路径和元数据），但它的 mesh 数据、子树内容都不在内存里。

payload_example.usda
#usda 1.0
def Xform "City" {
    # payload：先跳过，按需 load
    def Xform "DowntownDistrict" (
        prepend payload = @./downtown.usd@
    ) {
        # 这里可以放「不依赖 payload 内容」的元数据
        # 比如包围盒用于可视性剔除
        float3[] extentsHint = [(-100, 0, -100), (100, 50, 100)]
    }

    def Xform "SuburbanDistrict" (
        prepend payload = @./suburban.usd@
    ) { }
}
payload_load_demo.py
from pxr import Usd

# 打开时不加载任何 payload——非常快
stage = Usd.Stage.Open("city.usd", load=Usd.Stage.LoadNone)

# 可以遍历到 Prim 路径（但没有 mesh 数据）
downtown = stage.GetPrimAtPath("/City/DowntownDistrict")
print("prim exists:", downtown.IsValid())  # True
print("has payload loaded:", downtown.IsLoaded())  # False

# 用户聚焦到 downtown 区域时，按需 load
stage.Load("/City/DowntownDistrict")
print("now loaded:", downtown.IsLoaded())  # True
# 现在可以访问 mesh 数据
for child in downtown.GetChildren():
    print(child.GetPath())

什么时候用 reference，什么时候用 payload

USD 文档给出了清晰的指导原则：

官方建议 如果一个资产是「定义场景骨架」的关键内容（比如场景的 transform 层次、灯光布局），用 reference——这些内容总是需要立即可见。如果是「可以暂时不在内存里」的重型资产（高多边形 mesh、大型子场景），用 payload——这样 Stage 能以最小内存打开，然后实现「working set 管理」：只把当前需要的部分加载进来。

ConvertAsset 如何处理 payloads

ConvertAsset 的 _collect_asset_paths() 会同时扫描 references 和 payloads（以及 sublayers、variant 内部的 references / payloads）：

convert_asset/no_mdl/references.py（逻辑示意）
def _collect_asset_paths(stage):
    """扫描 stage 中所有 composition arc 里的 assetPath 条目。
    
    涵盖：
    - root layer 的 subLayers
    - 每个 Prim 上的 references（ListOp）
    - 每个 Prim 上的 payloads（ListOp）
    - 每个 VariantSet 的每个 variant 内的 references / payloads
    - clips（valueClips）
    
    返回六元组列表：
    (kind, holder, layer_dir, assetPath, prim_path, extra)
    """
    entries = []
    # 扫描所有 Prim
    for prim in stage.Traverse():
        # 检查 references
        refs = prim.GetMetadata("references")
        if refs:
            for item in refs.prependedItems + refs.appendedItems:
                entries.append(("reference", ("prim", str(prim.GetPath())),
                                layer_dir, item.assetPath, str(prim.GetPath()), item))
        # 检查 payloads（同理）
        payloads = prim.GetMetadata("payload")
        # ...
    return entries

对于 Processor 而言，reference 和 payload 是等价的——都需要被递归转换成 *_noMDL.usd 版本，然后在当前 layer 上把路径改写。

LIVERPS 里 payload < reference 从强度上看，payload 的 opinion 比 reference 弱（LIVERPS 中 P 排在 E 右边）。实践中，同一个 Prim 很少同时有 reference 和 payload——如果真的有，reference 的内容会覆盖 payload 的。通常工具会用其中一种，不混用。
