SM / CUDA core / warp / SIMT 执行模型

Streaming Multiprocessor（SM）是 GPU 里真正执行计算的处理单元。理解 SM 的内部结构——CUDA core、warp、SIMT（Single Instruction Multiple Threads）执行模型——是理解 GPU 并行计算性能边界的基础。本节基于 NVIDIA Turing 白皮书的数据。

From GAMES101 GAMES101 里的着色器是一段代码，在概念上"对每个 fragment 各执行一次"。但 GPU 并不是真的为每个 fragment 各启动一个独立线程——它把 32 个线程打包成一个 warp，让这 32 个线程同步执行同一条指令（SIMT），只是操作的数据不同。

Turing TU102 顶层结构

Turing TU102（RTX 2080 Ti / Quadro RTX 6000 使用的 GPU）的数字：

  • 6 个 GPC（Graphics Processing Cluster），每个 GPC 有 6 个 TPC
  • 36 个 TPC（Texture Processing Cluster），每个 TPC 含 2 个 SM
  • 72 个 SM，每个 SM 有 64 个 CUDA Core
  • 全 GPU 共 4608 个 CUDA Core、72 个 RT Core、576 个 Tensor Core

SM 内部结构

每个 Turing SM 分成 4 个处理块（Processing Block），每块包含：

  • 16 个 FP32 CUDA Core（浮点运算）
  • 16 个 INT32 Core（整数运算，Turing 新增）
  • 2 个 Tensor Core
  • 1 个 warp scheduler + 1 个 dispatch unit
  • 64 KB register file + L0 instruction cache

四个处理块共享一个 96 KB L1 data cache / shared memory，可按工作负载在 64 KB L1 + 32 KB shared 与 32 KB L1 + 64 KB shared 之间动态分配。

warp 与 SIMT

GPU 不是以单个线程为调度单位，而是以 32 个线程为一组——称为一个 warp。一个 warp 里所有线程在同一时钟周期执行同一条指令，但操作各自不同的数据（SIMT：Single Instruction, Multiple Threads）。

当某个 warp 需要等待内存访问时（如 texture fetch），warp scheduler 可以立即切换到另一个就绪的 warp，从而隐藏延迟（latency hiding）。这是 GPU 实现高吞吐量的根本机制——靠大量并发 warp 填满执行管道，而不是像 CPU 那样靠缓存和乱序执行来降低单条指令的延迟。

Turing 新增：独立整数管道

Turing 白皮书指出：现代 shader workload 里，每 100 条浮点指令中平均有 36 条整数指令（用于地址计算、索引等）。Pascal 及更早的架构里，整数指令和浮点指令共用同一条执行管道——整数指令运行时，FP32 管道会空闲。

Turing 在每个 FP32 Core 旁边加了一个独立的 INT32 Core，可以与浮点运算并发执行。这使得有效 FP32 吞吐量提升约 36%。结合统一的 L1/shared memory 设计，Turing SM 比 Pascal 每 CUDA core 的实际有效性能提升 50%（Turing whitepaper 数据）。

关键数字（Turing TU102）
  • 72 SM × 64 CUDA Core = 4608 CUDA Core
  • 72 SM × 8 Tensor Core = 576 Tensor Core
  • 72 SM × 1 RT Core = 72 RT Core（第 12 章主角）
  • 每 SM 96 KB L1/shared memory，全 GPU ~6.9 MB
  • RTX 2080 Ti（Founders Edition）：14.2 TFLOPS FP32 @ 1635 MHz boost
