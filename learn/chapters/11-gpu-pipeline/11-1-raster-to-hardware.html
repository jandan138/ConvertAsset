从 rasterization 回顾到硬件

GAMES101 告诉你什么是 rasterization、z-buffer，以及它们与 ray tracing 之间的根本差异。但软件实现和硬件实现之间隔着一道巨大的鸿沟——现代 GPU 的 Streaming Multiprocessor（SM）架构正是这道鸿沟的工程答案。本章带你从课堂 renderer 走到 NVIDIA Turing/Ampere 芯片，理解每块硬件对应流水线的哪个阶段。

From GAMES101 你在 GAMES101 里写过的光栅化 pipeline：vertex transform → triangle setup → rasterization → z-test → fragment shading → framebuffer。这条流水线在 GPU 里几乎一比一地对应：vertex shader → PolyMorph / raster engine → SM（fragment shader） → ROP（render output processor）→ frame buffer。

软件光栅化回顾

光栅化算法的核心是：对每个三角形，找出其覆盖的像素，然后对每个覆盖像素做深度测试（z-buffer test）并着色。这是一个"逐三角形"的操作，天然适合并行——不同三角形、不同像素完全独立，可以同时处理。

z-buffer 的精妙在于它把"可见性"变成了一个局部判断：不需要预排序，每个像素维护一个当前最近深度值，后来的片段做一次比较即可判断是否可见。这极大地简化了硬件实现。

rasterization vs ray tracing：两种根本不同的范式

GAMES101 里讲得很清楚：rasterization 是"以三角形为主体，问覆盖了哪些像素"；ray tracing 是"以像素为主体，向场景发射光线，问打到了哪个三角形"。

两者的本质区别在于光线与几何的查询方向：

  • 光栅化：$O(\text{triangles})$，每个三角形扫描它的包围盒，局部内存访问友好。
  • 光线追踪：每条光线需要查询整个场景的几何，随机访问全局数据结构（BVH），内存访问模式不规则。

正是这个差异决定了为什么 rasterization 在 GPU 上"自然地快"，而 ray tracing 在没有专用硬件时"天然地慢"——第 11-4 节会详细解释这一点。

GPU 作为硬件光栅化器

现代 GPU 把上述软件 pipeline 的每个阶段都用专用或可编程硬件来实现：

可编程阶段（Vertex Shader、Fragment Shader）在 SM（Streaming Multiprocessor）上运行——这是 GPU 最核心的处理单元，下一节会详细介绍它的内部结构。固定功能阶段（Raster Engine、ROP）做的事情对所有应用几乎相同，用专用硬件实现可以做到极高的吞吐量和功耗效率。

为什么光栅化天然适合 GPU

每个像素的 fragment shading 完全独立——不同像素之间不需要通信，没有数据依赖。这使得 GPU 可以用数千个 CUDA core 同时处理数千个 fragment，实现接近线性的吞吐量提升。Turing TU102 有 4608 个 CUDA core，意味着理论上能同时执行 4608 个 fragment 着色器线程。
