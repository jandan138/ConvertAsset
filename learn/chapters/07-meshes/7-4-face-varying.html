face-varying 的坑

faceVarying 是 USD 中最常见的 UV 插值模式，也是 USD→GLB 管线里最大的障碍。它允许同一个顶点在不同面上拥有不同的 UV 值（UV 缝合线、硬边法线都依赖这个特性），而 glTF 只支持 per-vertex 的属性——一个顶点只能有一套 UV。解决方案是顶点分裂（vertex splitting）：把拓扑"炸开"，每个面角点变成独立顶点。

为什么需要 faceVarying

想象一个正方体的某条棱——棱两侧的两个面有不同的 UV 展开岛（UV island）。棱上的顶点被两个面共享，但在两个面的 UV 空间里位置完全不同。如果强制用 vertex 插值，一个顶点只能绑定一个 UV，必然在某个面上产生纹理扭曲。

faceVarying 把数据绑定到"面角点"（face corner）而非顶点：对于 \(N\) 个三角形的网格，faceVarying 数组有 \(3N\) 条目，每个三角形的三个角点各自独立存储 UV，互不干扰。

与 glTF 的根本冲突

glTF 的数据模型里，accessor 以顶点为单位——每个顶点对应一套位置、法线、UV。"同一顶点两个 UV"在 glTF 规范里根本不存在。

因此，ConvertAsset 的 glb/usd_mesh.py 采用 mesh flattening（网格展平）策略：把整个 faceVertexIndices 重新解释为从 0 开始的连续顺序索引，同时把 points、normals、uvs 全部按这个顺序展开。展开后顶点数 = 原始面角点总数（= \(3N\) for 三角网格），每个顶点唯一。

$$ \text{顶点数}_{\text{展平后}} = \sum_i \text{faceVertexCounts}[i] = 3N \quad (\text{纯三角网格})$$

展平算法（vertex splitting）

以下是 glb/usd_mesh.py 的核心逻辑（精简版）：

glb/usd_mesh.py（mesh flattening 核心）
if needs_flattening:
    # 用 faceVertexIndices 作为"散射索引"展开 points 和 normals
    # 展开后：新顶点 i 对应原来的 points[faceVertexIndices[i]]
    points_np  = points_np[indices_np]   # shape: (3N, 3)
    if normals_np is not None:
        normals_np = normals_np[indices_np]

    # 展开 faceVarying UV
    if uv_indices is not None and len(uv_indices) == len(indices_np):
        # 有压缩索引：先解压再对齐
        uvs_np = uv_data[uv_indices]
    elif len(uv_data) == len(indices_np):
        # 无压缩索引：数据已与面角点对齐，直接用
        uvs_np = uv_data

    # 重生成索引：0, 1, 2, 3, ..., 3N-1（不再共享顶点）
    indices_np = np.arange(len(indices_np), dtype=np.uint32)

代价：顶点数膨胀

顶点分裂的代价是顶点数大幅增加。一个有 1000 三角形的典型网格，原本可能只有 600 个唯一顶点（共享率约 1.67），展平后变成 3000 个顶点。好消息是 GLB 使用 accessor + bufferView 的索引方式，引擎端的 GPU 顶点缓存仍能利用 post-transform cache 命中率，实际渲染开销增幅有限。

小结 faceVarying UV 对 USD 来说是"自然"的，对 glTF 来说是"非法"的。ConvertAsset 的解法是在 GLB 导出时无损地将网格展平：原有 USD 文件不受影响，只在写出 GLB 时在内存中做一次展开。这是一个正确且实用的工程折中。
