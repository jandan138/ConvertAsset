二进制解剖实操

理论说了那么多，不如亲手打开一个 GLB 文件用 xxd 和 Python 验证一遍。本节以 ConvertAsset 导出的一个简单资产为例，逐字节对照 9-3 的格式描述，再把每个 accessor 的字节范围与 glb/usd_mesh.py + glb/writer.py 的代码路径对应起来。

step 1：xxd 验证 magic 与版本

生成一个 GLB 之后，用 xxd 查看前 12 字节：

shell
./scripts/isaac_python.sh ./main.py export-glb scene_noMDL.usd --out out.glb

xxd out.glb | head -2

典型输出（小端）：

xxd 输出
00000000: 676c 5446 0200 0000 a842 0400 ...  glTF....
          ^^^^^^^^^^^^^^ ^^^^^^^^^^^^ ^^^^
          magic="glTF"   version=2   length=0x000442a8

67 6c 54 46 = ASCII "glTF"（注意小端存储下字节序与 0x46546C67 的字节排列）；接下来 4 字节 little-endian 读出 = 2 = version 2；再 4 字节是整个文件长度。

step 3：accessor 字节范围对应关系

ConvertAsset 的 glb/writer.py 按照如下顺序把数据打包进 BIN chunk（以一个简单网格为例）：

  1. POSITION accessor：float32 × 3 × N_verts 字节（顶点坐标）
  2. NORMAL accessor（若有）：float32 × 3 × N_verts 字节
  3. TEXCOORD_0 accessor（若有 UV）：float32 × 2 × N_verts 字节
  4. indices accessor：uint16 或 uint32 × N_indices 字节（根据顶点数选择类型）

所有块之间必须 4 字节对齐。下表是一个实际 3000 顶点、9000 面（三角）网格的典型布局：

accessor byteOffset（BIN内） byteLength 计算
POSITION 0 36000 3000 × 3 × 4
NORMAL 36000 36000 3000 × 3 × 4
TEXCOORD_0 72000 24000 3000 × 2 × 4
indices 96000 54000 9000 × 3 × 2（uint16，顶点数<65536）

数据流：usd_mesh.py → writer.py → GLB

综合前面各节，ConvertAsset 的 USD→GLB 数据流如下：

  1. glb/usd_mesh.py：从 UsdGeom.Mesh 提取 points / normals / UV（faceVarying UV 在此展平）→ numpy arrays。
  2. glb/usd_material.py：从 UsdPreviewSurface 提取 base color / metallic / roughness / textures。
  3. glb/converter.py：遍历 Stage 所有 Mesh，调用上述两个模块，组装 glTF JSON 字典。
  4. glb/writer.py：序列化 JSON + 打包 BIN → 写出 GLB header + chunks。
总结 解剖一个真实 GLB：magic "glTF" 在文件头 0 字节，JSON chunk 紧跟其后，BIN chunk 里存着 float32 顶点和 uint16/32 索引。ConvertAsset 的 writer.py 是这套格式的简洁实现——不到 200 行 Python，无第三方依赖，生成完全符合 glTF 2.0 规范的 GLB。
